### AC-3: Black — Color(0.2,0.2,0.2), 25x35, speed=150, HP=1, visual speed lines

```gd
class_name EnemyBlack
extends EnemyBase

var _last_position: Vector2 = Vector2.ZERO

func _ready() -> void:
    super()
    type = "black"
    body_color = Color(0.2, 0.2, 0.2)
    body_size = Vector2(25, 35)
    speed = 150.0
    max_health = 1
    health = max_health

func _physics_process(delta: float) -> void:
    _last_position = global_position
    super(delta)

func _draw() -> void:
    super()
    if velocity.length() > 50:
        var behind_offset: Vector2 = -Vector2.from_angle(rotation) * body_size.x * 0.6
        for i in range(3):
            var line_start := _last_position + behind_offset + Vector2.from_angle(rotation + PI/2) * (i - 1) * 8
            var line_end := line_start + (-velocity.normalized() * body_size.x * 0.4)
            draw_line(line_start - global_position, line_end - global_position, Color(0.4, 0.4, 0.4), 2.0)
```

**Status: PASS** (with one non-blocking timing concern)

**Non-blocking concern (NC-001):** `_last_position = global_position` at line 16 is assigned *before* `super(delta)` is called. The base class `EnemyBase._physics_process` then calls `move_and_slide()`, which updates `global_position`. So `_last_position` stores the position *after* movement rather than the position *before* this frame's movement.

However, the visual result is still correct: speed lines are drawn behind the enemy in the direction of travel. The one-frame lag is not perceptible given that the base movement system is already jerky (no acceleration smoothing — velocity = direction × speed each frame). This is a minor implementation detail that does not affect the visible result.

**Additional note (NC-002):** `super()` is called in `_draw()` even though `EnemyBase` does not define `_draw()`. This is a harmless no-op in GDScript but is unnecessary.

### AC-5: Boss — Color(1.0,0.84,0.0), 50x65, speed=100, HP=10, pulsing gold modulate

```gd
class_name EnemyBoss
extends EnemyBase

func _ready() -> void:
    super()
    type = "boss"
    body_color = Color(1.0, 0.84, 0.0)  # ✅ gold
    body_size = Vector2(50, 65)          # ✅ largest variant
    speed = 100.0                         # ✅
    max_health = 10                       # ✅
    health = max_health                   # ✅

func _process(delta: float) -> void:
    var t: float = sin(Time.get_ticks_msec() / 300.0) * 0.5 + 0.5
    modulate = Color(1.0, 0.84 + t * 0.16, 0.0 + t * 1.0, 1.0)
```

**Status: PASS**

The `_process` sine wave:
- `Time.get_ticks_msec() / 300.0` → argument to `sin()` increments at ~3.33 units/ms
- `sin(x) * 0.5 + 0.5` normalizes output to range `[0, 1]` with period ≈ 1885ms (~1.9 second full cycle)
- `modulate.g` oscillates `0.84 → 1.0` and `modulate.b` oscillates `0.0 → 1.0`
- This creates a smooth gold-to-bright-yellow pulsing effect — clearly visible and appropriate for a boss

**Informational note (NC-003):** Boss speed is fixed at `100.0`. The spec says "variable speed" which could imply speed variation during the fight (e.g., faster when low HP). The pulsing gold modulate is implemented, satisfying the "pulsing gold modulate" requirement. True dynamic speed variation would be a future enhancement.

### AC-6: Wave spawner can instantiate any variant by type name

```gd
func _create_enemy(type: String) -> EnemyBase:
    match type:
        "brown": return EnemyBrown.new()
        "black": return EnemyBlack.new()
        "war":   return EnemyWar.new()
        "boss":  return EnemyBoss.new()
        _:       return EnemyBrown.new()
```

**Status: PASS**

The `match` statement covers all four variant type strings. Return type is explicitly `EnemyBase`. The fallback returns `EnemyBrown.new()` which is safe. All `type` strings (`"brown"`, `"black"`, `"war"`, `"boss"`) match keys in `_point_values` dictionary, so score tracking in `_on_enemy_died` will work correctly for all variants.
