## Findings by Acceptance Criterion

### Criterion 1: Four enemy variant scripts exist matching the canonical spec
**Status: PASS**

The plan creates four distinct scripts:
- `scripts/enemy_brown.gd` — extends `EnemyBase`, type `"brown"`
- `scripts/enemy_black.gd` — extends `EnemyBase`, type `"black"`
- `scripts/enemy_war.gd` — extends `EnemyBase`, type `"war"`
- `scripts/enemy_boss.gd` — extends `EnemyBase`, type `"boss"`

Each sets `type` string correctly for wave-spawner score tracking via `_point_values`.

### Criterion 2: Brown — Color(0.6,0.4,0.2), 25x35, speed=80, HP=1
**Status: PASS**

`enemy_brown.gd` sets `@export` defaults exactly:
- `max_health = 1`
- `speed = 80.0`
- `body_color = Color(0.6,0.4,0.2)`
- `body_size = Vector2(25, 35)`

No special visual overrides needed; base class `Polygon2D` body + head is sufficient per plan.

### Criterion 3: Black — Color(0.2,0.2,0.2), 25x35, speed=150, HP=1, visual speed lines
**Status: PASS**

`enemy_black.gd` sets correct defaults (speed=150.0, body_size=Vector2(25,35), body_color=Color(0.2,0.2,0.2)).

Speed lines implemented as `_draw()` override:
- Calls `super` to render base body + head first
- Draws 3–4 grey `Color(0.4,0.4,0.4)` lines at ~60% body width
- Offset computed as `+body_size.x * 0.6` in opposite direction of `rotation`
- Uses `draw_line()` with `rotation` for directional orientation

**Non-blocking design note**: The speed-line coordinate approach is mathematically sound (lines positioned behind the body using rotation), but actual pixel-look depends on the base class `_create_visual()` head position. If visual results are off, the offset multiplier (0.6) or line length can be adjusted in implementation. No architectural concern.

### Criterion 4: War Horse — Color(0.8,0.2,0.2), 35x50, speed=60, HP=3, thicker body
**Status: PASS**

`enemy_war.gd` sets `@export` defaults exactly:
- `max_health = 3`
- `speed = 60.0`
- `body_color = Color(0.8,0.2,0.2)`
- `body_size = Vector2(35, 50)` — larger than brown/black, achieves "thicker" visual via base class Polygon2D rendering

**Non-blocking note**: Plan acknowledges war horse and brown use same Polygon2D shape — differentiated by color (red vs brown) and size (35x50 vs 25x35). Accepted per risk table.

### Criterion 5: Boss — Color(1.0,0.84,0.0), 50x65, speed=100 (variable), HP=10, pulsing gold modulate
**Status: PASS**

`enemy_boss.gd` sets correct defaults:
- `max_health = 10`
- `speed = 100.0`
- `body_color = Color(1.0,0.84,0.0)`
- `body_size = Vector2(50, 65)`

Pulsing implemented via `_process(delta)` override using:
```gdscript
modulate = Color(1.0,0.84,0.0) * (sin(Time.get_ticks_msec() * 0.003) * 0.3 + 0.85)
```
- Period: ~2 seconds (sin with 0.003 multiplier)
- Range: brightness scalar from ~0.55 to ~1.15 applied to gold base color
- Safe: `self.modulate` only, no child node pollution

**Non-blocking design note**: The damage flash in `EnemyBase.take_damage()` sets `modulate = Color(1.5, 1.5, 1.5)` and `_is_flashing = true`. During the 0.15s flash window, the boss's sine-pulse is overridden by the flash. Once `_is_flashing = false` and `modulate = Color.WHITE` is restored, the sine-pulse resumes. This interaction is acceptable per plan risk assessment. Implementation should ensure `_process()` does not reset `modulate` to a non-white value while `_is_flashing` is true (recommended mitigation in risk table).

### Criterion 6: Wave spawner can instantiate any variant by type name
**Status: PASS (with caveat on current state)**

The plan correctly replaces `EnemyBase.new()` + manual property assignment in `wave_spawner.gd._create_enemy()` with:
```gdscript
match type:
    "brown": return enemy_brown.new()
    "black": return enemy_black.new()
    "war": return enemy_war.new()
    "boss": return enemy_boss.new()
```

**Verification finding**: The current `wave_spawner.gd` at line 77 does `var enemy = EnemyBase.new()` and assigns `enemy.type = type` for score lookup. The `_point_values` dict already has all four keys ("brown", "black", "war", "boss"). The plan's update is the correct architectural fix.

**Note**: `enemy.type` is a `String` var on `EnemyBase` (line 12: `var type: String = "brown"`). The plan correctly sets `type` on each variant so `_on_enemy_died` can use `enemy.type` for score lookup via `_point_values.get(enemy.type, 10)`.
