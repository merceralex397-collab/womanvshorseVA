### [BLOCKER-1] smoke_test still classifies Godot class_name reload output as `syntax_error` — FINISH-VALIDATE-001 smoke test fails

**Severity:** correctness bug / blocker

**Location:** `.opencode/tools/smoke_test.ts`, `classifyCommandFailure` function

**Evidence:**

```
Command: godot4 --headless --path . --quit
exit_code: 0
failure_classification: syntax_error  ← should be tooling_parse_warning
blocked_by_permissions: false
Overall Result: FAIL
```

Actual smoke test output (`.opencode/state/smoke-tests/finish-validate-001-smoke-test-smoke-test.md`):
```
failure_classification: syntax_error
...
The smoke-test run failed because the smoke-test surface is not configured correctly.
```

**Root cause trace through the classification chain:**

The stderr for the Godot headless load validation command contains:
```
ERROR: Failed to load script "res://scripts/wave_spawner.gd" with error "Parse error".
   at: load (modules/gdscript/gdscript.cpp:2907)
```

This line matches `isGodotFatalDiagnosticOutput` at line 582–583:
```typescript
function isGodotFatalDiagnosticOutput(output: string): boolean {
  return /SCRIPT ERROR:.*(?:not declared in the current scope|not found in base self)/i.test(output)
    || /Failed to load script "res:\/\//i.test(output)   // ← this matches
}
```

The stderr ALSO contains all three patterns for `isClassNameReloadParseWarning` (lines 586–591):
- `exitCode === 0` ✓
- `stderr.includes("Could not parse global class")` ✓
- `stderr.includes("Could not resolve class")` ✓ (EnemyBrown, EnemyBlack, EnemyWar, EnemyBoss)

**However:** The `Failed to load script "res://scripts/wave_spawner.gd"` line in the combined `output` (stdout + "\n" + stderr) matches `isGodotFatalDiagnosticOutput`'s second pattern unconditionally — it fires for ANY `Failed to load script "res://..."` message, regardless of exit code.

The fix in the implementation artifact correctly orders `isClassNameReloadParseWarning` before `isGodotFatalDiagnosticOutput` in the exit_code===0 branch. But `isGodotFatalDiagnosticOutput` also matches this stderr independently. For the class_name reload tooling artifact to be properly classified as `tooling_parse_warning`, the `isClassNameReloadParseWarning` check must fire first AND prevent `isGodotFatalDiagnosticOutput` from overriding it.

**The `isClassNameReloadParseWarning` check must additionally ensure that `isGodotFatalDiagnosticOutput` does NOT also match**, OR `isGodotFatalDiagnosticOutput` must be updated to NOT fire on `Failed to load script` when the context is a class_name reload parse artifact with exit_code===0.

**Required fix (one of these approaches):**

1. **Option A — Guard `isGodotFatalDiagnosticOutput` in exit_code===0 path:** After `isClassNameReloadParseWarning` returns `tooling_parse_warning`, skip `isGodotFatalDiagnosticOutput` entirely:
   ```typescript
   if (args.exitCode === 0) {
     if (isClassNameReloadParseWarning(args.stderr, args.exitCode)) return "tooling_parse_warning"
     if (isGodotFatalDiagnosticOutput(output)) return "syntax_error"
     return undefined
   }
   ```
   This is the intended fix per the implementation artifact. It works IF `isClassNameReloadParseWarning` fires first. But `isGodotFatalDiagnosticOutput` ALSO matches this stderr, so if the helper returns `false` for any reason, `syntax_error` wins.

2. **Option B — Add an exclusion to `isGodotFatalDiagnosticOutput`:** Exclude `Failed to load script` matches when exit_code===0 AND the class_name reload pattern is present:
   ```typescript
   function isGodotFatalDiagnosticOutput(output: string, exitCode?: number): boolean {
     if (/Failed to load script "res:\/\//i.test(output) && exitCode === 0) {
       // These are class_name reload tooling artifacts in Godot headless mode
       return false
     }
     return /SCRIPT ERROR:.*(?:not declared in the current scope|not found in base self)/i.test(output)
       || /Failed to load script "res:\/\//i.test(output)
   }
   ```
   And update the call site: `isGodotFatalDiagnosticOutput(output, args.exitCode)`.

3. **Option C — Combine the check:** In the exit_code===0 branch, check both conditions together before returning anything:
   ```typescript
   if (args.exitCode === 0) {
     const isClassNameReload = isClassNameReloadParseWarning(args.stderr, args.exitCode)
     const isFatal = isGodotFatalDiagnosticOutput(output)
     if (isClassNameReload) return "tooling_parse_warning"
     if (isFatal) return "syntax_error"
     return undefined
   }
   ```
   Same result as Option A — but explicit.

The core issue: `isClassNameReloadParseWarning` and `isGodotFatalDiagnosticOutput` are not mutually exclusive for the `Failed to load script "res://..."` pattern. Both can fire on the same stderr. The ordering in the implementation artifact is correct IF the helper fires first, but the smoke test shows `syntax_error`, meaning `isClassNameReloadParseWarning` is returning `false` or the code is not yet live.
